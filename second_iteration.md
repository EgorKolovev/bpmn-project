# ТЗ: Итерация 2 — Улучшение генерации BPMN-диаграмм

## Контекст

МВП принят заказчиком. После тестирования он выдвинул три требования:
1. Поддержка русского языка в генерируемых диаграммах
2. Генерация сложных (нелинейных) BPMN-процессов с условиями, ветвлениями и возвратами
3. Поддержка дорожек (Swimlanes/Lanes) — привязка каждого шага к роли исполнителя

### Текущее состояние системы

- LLM: Gemini 3.1 Flash Lite (низкая модель, малый reasoning)
- Промпты: `ml/app/prompts.py` — три промпта: classify, generate, edit
- BPMN генерируется напрямую LLM-ом, затем проходит валидацию и пост-обработку
- Поддерживаются: `task`, `exclusiveGateway`, `parallelGateway`, `startEvent`, `endEvent`, `sequenceFlow`
- **Не поддерживаются**: lane/pool/swimlane, условные переходы, циклы, русский язык в лейблах

---

## Задача 1: Русский язык в BPMN-диаграммах

### Проблема
Все названия задач, событий и шлюзов генерируются на английском. Заказчик работает в русскоязычной среде.

### Решение
Промпт-инжиниринг в ML-сервисе: добавить явную инструкцию в `SYSTEM_PROMPT_GENERATE` и `SYSTEM_PROMPT_EDIT`, предписывающую использовать язык входного текста для названий всех элементов диаграммы. Если описание дано на русском — все `name` атрибуты и `<bpmn:conditionExpression>` должны быть на русском.

**Конкретно:**
- Добавить в оба промпта (generate + edit) правило: *"Use the same language as the user's input for all element names. If the input is in Russian, all task names, gateway names, event names, and condition labels must be in Russian."*
- Добавить в classify-промпт поддержку русскоязычного входа (сейчас он может отфильтровывать русский текст как нерелевантный)
- Проверить, что классификатор (`/classify`) корректно принимает русскоязычные описания бизнес-процессов

### Затронутые модули
- `ml/app/prompts.py` — промпты GENERATE, EDIT, CLASSIFY

## Задача 2: Сложные нелинейные BPMN-процессы

### Проблема
LLM генерирует примитивные линейные цепочки. Отсутствуют: условные переходы, циклы, возвраты к предыдущим шагам, именованные ветви.

### Причины
1. Слабая модель (Gemini Flash Lite — оптимизирована для скорости, не для reasoning)
2. Промпт не содержит явных инструкций по применению сложных конструкций и не имеет few-shot примеров

### Решение

#### 2.1 Обновление промпта генерации

Переписать `SYSTEM_PROMPT_GENERATE` с явными правилами применения сложных конструкций и добавить few-shot пример:

**Новые правила в промпте:**
- *"If the description implies a decision or branching ('if', 'либо', 'в случае', 'при условии', 'иначе') — use `exclusiveGateway` with named outgoing sequence flows (set `name` attribute on each flow with the condition label)"*
- *"If the description implies a loop or retry ('повторить', 'вернуться', 'если не прошло — заново') — model it as a cycle: exclusiveGateway → task → back to previous gateway"*
- *"Condition labels on sequenceFlow must be meaningful text, not 'Yes/No', e.g. 'Одобрено', 'Отклонено', 'Требует доработки'"*
- *"Always prefer richer process models over flat linear chains. A 3-step description can still have a conditional branch if it makes business sense."*
- *"Include `<bpmn:conditionExpression>` inside sequenceFlow elements that represent conditional paths"*

**Few-shot пример** (встроить в промпт как образец):
```
User: "Процесс согласования договора: менеджер создаёт заявку, юрист проверяет,
если замечания есть — возвращает на доработку, если нет — директор подписывает."

Expected BPMN structure:
startEvent → task(Создать заявку) → task(Проверка юристом) →
exclusiveGateway(Замечания?)
  → [Есть замечания] → task(Доработка) → back to task(Проверка юристом)
  → [Нет замечаний] → task(Подпись директора) → endEvent
```

#### 2.2 Обновление модели LLM

Сменить модель с `gemini-3.1-flash-lite-preview` на модель с улучшенным reasoning:
- **Вариант A (рекомендуемый)**: `gemini-2.0-flash` или `gemini-2.5-flash` — баланс цены и качества
- **Вариант B**: `gemini-2.5-pro` — максимальный reasoning, но дороже
- Скорректировать `DAILY_SPEND_LIMIT_USD` в конфиге с учётом новой ценовой модели

#### 2.3 Обновление валидатора

Расширить `ml/app/validator.py`:
- Добавить проверку наличия `name` атрибута на outgoing-потоках из `exclusiveGateway` (предупреждение, не ошибка)
- Добавить валидацию: если есть `exclusiveGateway` с 2+ исходящими потоками, хотя бы один должен иметь `conditionExpression` или `name`

#### 2.4 Обновление пост-обработки

Убедиться, что `ensure_incoming_outgoing()` в `ml/app/bpmn_fix.py` корректно обрабатывает сложные графы с циклами (обратными sequenceFlow).

### Затронутые модули
- `ml/app/prompts.py` — полная переработка SYSTEM_PROMPT_GENERATE
- `ml/app/config.py` — смена модели, корректировка бюджета
- `ml/app/validator.py` — расширение правил валидации

## Задача 3: Поддержка ролей через Swimlanes (Lanes/Pools)

### Проблема
Заказчик требует визуального разделения процесса по ролям исполнителей. В BPMN это реализуется через конструкцию `laneSet` → `lane`, где каждая дорожка содержит задачи конкретного исполнителя.

### Ожидаемый результат
Диаграмма с горизонтальными дорожками, подпись роли располагается **вертикально слева** от дорожки (как на скриншоте заказчика: «Менеджер из Юнита» — верхняя дорожка, «Контрактный менеджер» — нижняя). Каждая задача размещена в дорожке своего исполнителя. Потоки могут пересекать границы дорожек (cross-lane sequenceFlow — это норма BPMN).

### Структура BPMN с Lanes

```xml
<bpmn:process id="Process_1">
  <bpmn:laneSet id="LaneSet_1">
    <bpmn:lane id="Lane_1" name="Контрактный менеджер">
      <bpmn:flowNodeRef>Task_1</bpmn:flowNodeRef>
      <bpmn:flowNodeRef>Task_3</bpmn:flowNodeRef>
    </bpmn:lane>
    <bpmn:lane id="Lane_2" name="Менеджер юнита">
      <bpmn:flowNodeRef>Task_2</bpmn:flowNodeRef>
    </bpmn:lane>
  </bpmn:laneSet>
  <!-- обычные flow-элементы -->
  <bpmn:task id="Task_1" name="Создать заявку">...</bpmn:task>
  ...
</bpmn:process>
```

### Решение

#### 3.1 Обновление промпта генерации

Добавить в `SYSTEM_PROMPT_GENERATE` раздел про Lane:
- *"If the description mentions different roles, departments, or actors performing different steps — group tasks into lanes using `<bpmn:laneSet>` inside the process element."*
- *"Each `<bpmn:lane>` has a `name` attribute (the role name) and contains `<bpmn:flowNodeRef>` for each flow node belonging to that role."*
- *"Every flow node (task, gateway, event) must appear in exactly one lane's `<bpmn:flowNodeRef>` list."*
- *"startEvent and endEvent should also be assigned to the most relevant lane."*
- *"If roles are not mentioned in the description — do NOT generate lanes; generate a flat process."*
- Добавить few-shot пример генерации с lanes

#### 3.2 Обновление пост-обработки (bpmn_fix.py)

Добавить функцию `ensure_lane_refs()`:
- После генерации проверить, что все flow-узлы процесса присутствуют в `<bpmn:flowNodeRef>` какого-либо lane (если laneSet существует)
- Если узел не привязан к lane — добавить его в первый lane как fallback
- Логика идентична `ensure_incoming_outgoing()` — идемпотентна, работает на уровне XML

#### 3.3 Обновление валидатора (validator.py)

Добавить правила:
- Если в процессе есть `laneSet` — проверить, что каждый flow-узел упомянут ровно в одном `lane` через `flowNodeRef`
- Если узел упомянут в нескольких lanes — это ошибка валидации (запустить retry)

#### 3.4 Фронтенд

bpmn-js уже нативно поддерживает отрисовку lanes. Дополнительных изменений на фронтенде **не требуется** — auto-layout из `bpmn-auto-layout` также поддерживает lanes. Нужно убедиться, что версия библиотеки (`v1.2.0`) корректно раскладывает процессы с lanes (проверить вручную).

### Затронутые модули
- `ml/app/prompts.py` — добавить lane-инструкции в GENERATE и EDIT промпты
- `ml/app/bpmn_fix.py` — добавить `ensure_lane_refs()`
- `ml/app/validator.py` — добавить валидацию lane-refs

## Задача 4: Тестирование на бенчмарках заказчика

### Контекст
Заказчик предоставил первый бенчмарк: документ **«Служебная командировка»** — подробная текстовая инструкция по оформлению командировок + две эталонные BPMN-схемы ("Fly" и "Fly for CEO and heads"). Это реальный пример того, что система должна уметь генерировать.

**Что демонстрирует бенч:**
- Сложный нелинейный процесс с множеством ветвлений (несколько exclusiveGateway по ходу)
- Три чётко выраженные роли: **Сотрудник**, **Бухгалтерия/Кадры**, **Руководитель отдела** — оформленные как swimlanes
- Условные переходы с именованными ветвями («Да»/«Нет», «по своей вине»/«не по своей вине»)
- Полностью русскоязычные подписи
- Вариативность: упрощённая схема (Fly) и расширенная с доп. ролями (Fly for CEO and heads)

**Что демонстрирует скриншот (эталон визуала swimlanes):**
- Дорожки расположены **горизонтально**, подпись роли — вертикально **слева** от дорожки
- Роли в примере: «Менеджер из Юнита» (верхняя дорожка) и «Контрактный менеджер» (нижняя)
- Внутри каждой дорожки — полноценные гейтвеи, задачи, события
- Потоки свободно пересекают границы дорожек (cross-lane sequenceFlow)

### Задачи
- Прогнать текст из документа «Служебная командировка» через обновлённую систему
- Сравнить результат с эталонными схемами "Fly" и "Fly for CEO and heads" по критериям:
  - Все роли распознаны и оформлены как lanes
  - Ветвления соответствуют описанию (опоздал по своей вине / не по своей вине, срок менее 3 дней и т.д.)
  - Все подписи на русском языке
  - XML корректно открывается в Camunda Modeler
- Зафиксировать результаты в сравнительной таблице (для отчёта по критерию 3 ТЗ проекта)
- Запросить у заказчика дополнительные бенчи при необходимости
